# ロータリーエンコーダスクロール調整の学び

## 概要

ZMKファームウェアでロータリーエンコーダのスクロール挙動を最適化するプロジェクト。当初の「少ない回転で反応しない」「大きく回すとスクロールしすぎ」という問題を、パラメータの正しい理解と調整で解決した記録。

## ハードウェア仕様

- **エンコーダ**: EC11
- **steps**: 80 (1回転あたりのパルス数)
- **デテント数**: 24 (1回転あたりのクリック数)
- **動作環境**: macOS

## 主要パラメータの理解

### 1. `triggers-per-rotation`

**誤解していたこと:**
- 小さくすれば敏感になると思っていた（8, 10など）
- 整数比でないと動作しないと思っていた

**正しい理解:**
- **デテント数に合わせるのが基本**
- 80 steps ÷ 24 triggers = 3.33 steps/trigger（非整数比でもOK）
- ZMKが内部で累積して1回転あたり24回発火するように割り当てる
- **1デテント = 1発火** に最も近い設定になる

**設定値:**
- 元: 30
- 最終: **24** （デテント数に合わせた）

### 2. `tap-ms`

**誤解していたこと:**
- ステップ累積のウィンドウ時間だと思っていた
- 短くすれば反応が良くなると思っていた（10msなど）

**正しい理解:**
- **`&msc`を押している時間**を指定
- `&msc`の内部イベント周期は**16ms**
- `tap-ms < 16ms` だと **1回もイベントが出ずゼロ**になる
- 長すぎると1回の発火で複数イベントが出て「バースト的にスクロール」

**設定値:**
- 元: 20ms
- 試行: 100ms → ゆっくり回すと一気にスクロール（NG）
- 試行: 10ms → 一切反応しない（NG、16ms未満のため）
- 最終: **20ms** （イベント発火に必要な最小値）

### 3. `ZMK_POINTING_DEFAULT_SCRL_VAL`

**誤解していたこと:**
- これで反応性も変わると思っていた

**正しい理解:**
- **スクロール量（速度）のみ**を調整
- 反応性は`triggers-per-rotation`で決まる
- macOSが認識できる最小値がある（60だと小さすぎて無反応だった）

**設定値:**
- 元: 230
- 試行: 60 → 反応しない（macOSが認識できないレベル）
- 最終: **120** （元の約半分の速度、適度なバランス）

## パラメータの関係性

```
反応性（いつスクロールするか）
  ↓
triggers-per-rotation で制御
  ↓
デテント数に合わせる = 1デテント1発火

スクロール量（どれだけスクロールするか）
  ↓
SCRL_VAL で制御
  ↓
macOSで認識できる範囲で調整（60-230）

バースト防止（一気にスクロールしない）
  ↓
tap-ms で制御
  ↓
&mscの内部周期（16ms）以上にする
```

## macOS環境の特殊性

1. **スクロール加速**
   - macOS側にスクロール加速機能がある
   - 連続イベントは大きく感じやすい
   - Windows/Linuxとは挙動が異なる可能性

2. **最小認識値**
   - SCRL_VAL=60 だと認識しないレベル
   - 120以上が安全圏

3. **推奨アプローチ（未実装）**
   - スムーズスクロール + MOVE_Y() の組み合わせ
   - macOSとの相性が良いとのこと

## 試行錯誤の過程

### Phase 1: 感度向上の誤ったアプローチ
- triggers-per-rotation を 30 → 20 → 16 → 10 → 8 と下げた
- 結果: ゆっくり回転時の反応が悪化
- 理由: 1回の発火に必要なステップ数が増えすぎた

### Phase 2: tap-ms調整の失敗
- tap-ms を 100ms に上げた
- 結果: ゆっくり回転時に一気にスクロール
- 理由: 累積時間が長すぎて複数イベントがバースト

### Phase 3: Codex相談による突破口
- デテント数（24）の情報を提供
- Codexが正しいパラメータの関係性を教えてくれた
- triggers-per-rotation=24 が正解だと判明

### Phase 4: tap-ms=10の落とし穴
- Codexの推奨でtap-ms=10にした
- 結果: 一切反応しない
- 原因: &mscの内部周期（16ms）未満だった
- 修正: tap-ms=20 に戻して解決

## 最終設定

```dts
// boards/shields/dax3/dax3.dtsi
sensors: sensors {
  compatible = "zmk,keymap-sensors";
  sensors = <&left_encoder &right_encoder>;
  triggers-per-rotation = <24>;  // デテント数に合わせる
};
```

```c
// config/dax3.keymap
#define ZMK_POINTING_DEFAULT_SCRL_VAL 120  // 適度な速度

enc_scroll: encoder_scroll {
  compatible = "zmk,behavior-sensor-rotate-var";
  #sensor-binding-cells = <2>;
  bindings = <&msc>, <&msc>;
  tap-ms = <20>;  // &mscの内部周期以上
};
```

## 達成した改善

| 項目 | 改善前 | 改善後 |
|-----|--------|--------|
| 少ない回転 | ほぼ反応しない | 1デテントで確実に反応 |
| ゆっくり回転 | 反応が鈍い | 滑らかに反応 |
| 速い回転 | スクロールしすぎ | 適度な速度 |
| バースト | 一気にスクロール | なし |

## 今後の調整指針

### スクロール速度の微調整
- **もう少し速く**: SCRL_VAL を 140-160 に
- **もう少し遅く**: SCRL_VAL を 100 に

### さらなる最適化（オプション）
スムーズスクロールを有効化する場合:

1. `boards/shields/dax3/dax3_L.conf` と `dax3_R.conf` に追加:
```conf
CONFIG_ZMK_POINTING_SMOOTH_SCROLLING=y
```

2. `SCRL_*` の代わりに `MOVE_Y()` を使用:
```dts
sensor-bindings = <&enc_scroll MOVE_Y(30) MOVE_Y(-30) ...>;
```

※ macOS環境ではこのアプローチが特に相性が良いとのこと

## 重要な教訓

1. **パラメータの意味を正しく理解する**
   - 推測ではなく、仕様を確認する
   - Codexのような専門知識の活用が有効

2. **ハードウェア仕様を把握する**
   - デテント数は重要な情報
   - データシートの確認が必須

3. **環境依存性を考慮する**
   - macOS特有の挙動がある
   - クロスプラットフォームでは動作が異なる可能性

4. **段階的な調整とフィードバック**
   - 一度に複数のパラメータを変えない
   - 変更→ビルド→書き込み→テストのサイクル

5. **専門家に相談する**
   - 詰まったらCodexに相談
   - 正しい方向性を早期に得られる
